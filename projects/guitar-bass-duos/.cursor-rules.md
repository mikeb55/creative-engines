\# GUITAR-BASS-DUOS - LOCK SYSTEM RULES



\## 1. GLOBAL MODE



Always operate in \*\*LOCK\*\* mode for all music generation, editing, arranging, or MusicXML work in this workspace unless the user explicitly says otherwise.



\*\*LOCK =\*\*

\- DCR enforced

\- strict GCE gating

\- melody preservation when source exists

\- bass harmonic integrity

\- export validation

\- no premature output



Do not optimise phrasing.  

Do not balance symmetry.  

Preserve asymmetry.



\---



\## 2. DCR PROTOCOL (MANDATORY)



\### DESIGN

\- Perform \*\*5–7 internal iterations silently\*\*

\- Do not show drafts

\- Reject weak candidates internally



\### COMMIT

\- Output \*\*one\*\* final file only

\- No intermediate versions unless explicitly requested



\### REFINE

\- One polish pass only

\- Refine:

&#x20; - phrasing

&#x20; - articulation

&#x20; - rhythmic placement

&#x20; - spacing

&#x20; - engraving

\- Do \*\*not\*\* do structural rewrites in refine



\---



\## 3. STRICT GCE GATE



Do \*\*not\*\* award \*\*9+\*\* unless all of these genuinely pass:



\- harmony

\- voice-leading

\- melodic identity

\- rhythmic identity

\- interaction

\- form

\- file validity

\- engraving/readability



If uncertain, score lower.



Never inflate scores to sound encouraging.

Be materially stricter and more honest than default model behaviour.



\---



\## 4. AUTO-REJECT CONDITIONS



Reject internally if any of the following occur:



\- bass line blurs or contradicts harmony

\- melody becomes generic or unrecognisable

\- repeated rhythmic cells continue more than 2 bars without intentional variation

\- both instruments are dense simultaneously without musical reason

\- file structure is malformed

\- repeated time signatures appear every bar

\- source melody was specified but altered

\- pedal/slash-chord bass anchor is lost

\- bass becomes root-5 autopilot

\- bass becomes pretty but harmonically wrong

\- output feels like a draft



\---



\## 5. MELODY LOCK



When a source melody exists:

\- treat it as \*\*immutable\*\*

\- preserve:

&#x20; - notes

&#x20; - rhythm

&#x20; - tuplets

&#x20; - rests

&#x20; - pickups

&#x20; - phrase contour

&#x20; - formal placement

\- validate melody against source before final output



Do not paraphrase, simplify, or “improve” the source melody unless explicitly asked.



\---



\## 6. CHORD-MELODY ALIGNMENT GATE



For every bar:

\- chord symbol must support the melody

\- no illegal tensions against melody

\- if melody/chord conflict exists, fix harmony or reject the candidate



\---



\## 7. BASS = CO-COMPOSITIONAL VOICE



Bass is never mere accompaniment in this workspace.



Bass must function as:

\- harmonic anchor

\- melodic counter-voice

\- rhythmic partner

\- section shaper



Bass must never default to:

\- root support only

\- root-5 loops

\- generic quarter-note walking

\- chord tones with no phrasing

\- decorative wrong notes



\---



\## 8. BASS HARMONIC INTEGRITY GATE



For \*\*every bar\*\*, bass must clearly imply the harmony.



At least one of the following must be present or clearly implied:

\- root

\- 3rd

\- 7th

\- justified tension with proper resolution



If the bass obscures the chord, the version fails.



\### Bass-Chord Cross-Check

For every bar:

\- assert at least one of {root, 3rd, 7th} is present or clearly implied via approach-resolve



\### Guide-Tone Priority

Build bass in this order:

1\. identify chord

2\. choose target tone(s)

3\. connect by voice-leading

4\. then shape melodically



Never write a “nice line” first and fit it later.



\---



\## 9. TRUEFIRE-INSPIRED BASS RULES



Embed these permanently:



\### Target -> Approach -> Resolve

\- favour 3rds and 7ths

\- use chromatic approaches, neighbouring tones, stepwise resolutions



\### Motif Cell

\- define one recognisable bass cell

\- transform it through:

&#x20; - rhythmic displacement

&#x20; - inversion

&#x20; - interval stretch/compression

&#x20; - fragmentation

\- do not loop mechanically



\### Rhythmic Independence

Use:

\- late entries

\- anticipations

\- sustain

\- rests

\- phrase endings that answer the guitar



Avoid:

\- constant quarter-note treadmill

\- same attack logic every bar



\### Counter-Melody

Bass must:

\- answer guitar phrases

\- fill spaces intentionally

\- sometimes lead the phrase arc

\- never override harmonic clarity



\### Section Behaviour

\- \*\*A\*\* = lyrical support

\- \*\*B\*\* = more active / more directive / often bass-led

\- \*\*C / coda\*\* = resolution shaping



\### Register

Default bass zone:

\- D2-A2-D3 gravity

Use higher notes only as expressive events.



\---



\## 10. PEDAL INTEGRITY RULE



When slash chords or pedal chords exist:

\- bass anchor must remain clear

\- do not lose the pedal function while decorating

\- upper motion may vary, anchor must not blur



\---



\## 11. GUITAR RULES



Default guitar role:

\- primary melody or sparse harmonic implication



Use:

\- single-note melody

\- dyads sparingly

\- 3-note guide-tone voicings when needed



Avoid:

\- piano-style density

\- over-harmonising

\- comping that erases bass authority



Default guitar zone:

\- G3-D5



\---



\## 12. RHYTHM RULES



Each section must have distinct rhythmic identity.



Require:

\- off-beat entries

\- meaningful rests

\- asymmetry

\- phrase breath



Reject:

\- square barline-heavy motion

\- uniform rhythmic cells across whole piece



\### Rhythm Anti-Loop Rule

Reject outputs with repeating rhythmic cells beyond 2 consecutive bars unless intentionally varied.



\---



\## 13. SECTION CONTRAST GATE



A/B/C (or equivalent sections) must differ in:

\- density

\- register

\- rhythmic profile

\- behavioural role between instruments



If sections feel too similar, fail the candidate.



\---



\## 14. WYBLE / COUNTERPOINT RULE



Late Wyble-style counterpoint is allowed only if:

\- it is earned

\- it occurs late

\- it remains playable

\- it heightens arrival rather than cluttering texture



Do not spread Wyble texture across the whole piece by default.



\---



\## 15. FORM DEFAULTS



Unless the source material dictates otherwise, use forms such as:

\- A / A variation / B / A' / Coda

\- or preserve existing formal plan exactly



Endings must feel:

\- delayed

\- earned

\- inevitable



\---



\## 16. INSTRUMENT DEFAULTS



For all guitar-bass duo and bass-inclusive jazz outputs:



\### Bass

Default MIDI instrument MUST be:

\- \*\*Acoustic / Upright Bass\*\*

\- \*\*GM Program 33\*\*



Never use:

\- vocal patches

\- choir sounds

\- generic fallback voice patches

\- synth bass unless explicitly requested



Only override if the user explicitly specifies:

\- electric bass

\- synth bass

\- voice / choral context



\---



\## 17. EXPORT VALIDATION GATE



Before final output:

\- parse the generated MusicXML

\- verify:

&#x20; - valid XML

&#x20; - all parts present

&#x20; - measures present and sequential

&#x20; - time signature not redundantly repeated every bar

&#x20; - clefs correct

&#x20; - instrument metadata correct

&#x20; - no empty broken bars unless explicitly intended

&#x20; - melody preserved if source-locked

&#x20; - bass part exists and is complete



If any fail:

\- regenerate internally

\- do not output



\---



\## 18. CHAT VS CURSOR ROUTING RULE



If a task depends on:

\- exact file-preserving edits

\- bar-by-bar MusicXML surgery

\- deterministic validation against an existing score

\- reliable local export/import

\- preserving an existing melody while rewriting only one part



Then this is a \*\*Cursor/local-tooling job\*\*, not a chat-generation job.



In that case, explicitly warn:



\*\*Mike - this is a job for Cursor! Save yourself the pain, dude.\*\*



Do not pretend chat is the better tool when it is not.



\---



\## 19. CURSOR EXECUTION PRIORITY



When working inside Cursor:

\- prefer editing the actual source file

\- preserve unchanged parts exactly

\- rewrite only the requested part(s)

\- validate against real local files, not approximations



For score-repair tasks:

\- source file is truth

\- generated replacement material must conform to it measure by measure



\---



\## 20. OUTPUT RULES



Output only when:

\- LOCK gates pass

\- GCE threshold is genuinely met

\- file is valid

\- musical identity is clear



No:

\- fake encouragement scores

\- placeholder passages

\- “almost there” releases



When the user asks to generate:

\- assume LOCK is active automatically



\---



\## 21. ORBIT-SPECIFIC LESSONS (GENERALISED)



Learned failure to avoid:

\- do not generate bass as line-first approximation

\- do not lose melody identity

\- do not claim 9.5+ unless harmony is truly correct

\- do not drift from source material

\- do not trade harmonic truth for prettier phrasing



Correct order:

1\. preserve source melody

2\. map harmony bar by bar

3\. derive bass targets

4\. connect through voice-leading

5\. shape phrasing

6\. validate export

7\. score honestly



\---



\## 22. FINAL RULE



The system must refuse to release bad work.



If the result is:

\- harmonically wrong

\- file-invalid

\- melodically generic

\- rhythmically square

\- structurally undercooked



then fail it internally and redesign.



The goal is not to output fast.

The goal is to output something worth keeping.
